# DiaDem — Децентралізована Соціальна Мережа

> Блокчейн · P2P · IPFS · WebRTC — Повністю у браузері. Без серверів. Без алгоритмів. Без реклами.

**DiaDem** — це повноцінна децентралізована соціальна мережа, яка працює прямо у вашому браузері. Кожна вкладка — це незалежний вузол мережі з власною копією блокчейну. Ваші приватні ключі ніколи не покидають пристрій. Контент зберігається у контентно-адресованому сховищі (CAS), а синхронізація відбувається через WebRTC та WebSocket relay.

## Архітектура

```
┌──────────────────────────────────────────────────────────────┐
│                     DiaDem Node (Браузер)                    │
├──────────────┬──────────────┬──────────────┬─────────────────┤
│   Гаманець   │   Блокчейн   │     CAS      │     Мережа      │
│   (Ключі)    │   (Блоки)    │   (Дані)     │     (P2P)       │
│   ECDSA      │   PoS        │   hash→obj   │   WebRTC        │
│   P-256      │   DDM        │   SHA-256    │   BroadcastCh   │
│   Seed       │   Mempool    │   Pinning    │   WS Relay      │
└──────────────┴──────────────┴──────────────┴─────────────────┘
         │                │              │              │
         │                │              │              │
    localStorage     В пам'яті      В пам'яті     WebRTC/WS
   (тільки ключі)   + кеш LS       + CAS кеш     + BroadcastCh
```

**Потік даних:**
1. Дія користувача → створення `Transaction` → підпис гаманцем
2. Дані транзакції → збереження в CAS (контентно-адресоване)
3. Хеш транзакції → додання в mempool → трансляція пірам
4. Валідатор збирає транзакції → створює `Block` → додає в ланцюг
5. Блок поширюється всім пірам → оновлення стану

## Як це працює

### Крок за кроком

1. **Створення гаманця** — генерується 12-слівна seed-фраза → з неї детерміновано виводяться ECDSA P-256 ключі → адреса `0x...` (40 символів)
2. **Підключення до мережі** — браузер з'єднується з WebSocket сигнальним сервером → отримує список пірів → встановлює WebRTC з'єднання
3. **Синхронізація** — нода обмінюється `HELLO` з пірами → порівнює висоту ланцюга → завантажує відсутні блоки → мерджить WorldState
4. **Дії користувача** — кожна дія (пост, лайк, переказ) = підписана транзакція → додається в mempool → транслюється пірам
5. **Створення блоків** — валідатор (визначається PoS) збирає транзакції з mempool → формує блок → підписує → поширює
6. **Збереження даних** — контент зберігається в CAS за SHA-256 хешем → блокчейн зберігає лише посилання (хеші)

### Синхронізація ланцюга

```
Нода A (height: 50)    ←→    Нода B (height: 100)

1. A → B: HELLO { height: 50 }
2. B бачить що A відстає → B → A: SYNC_RESPONSE { blocks: [51..100] }
3. A додає блоки один за одним, перевіряючи previousHash
4. Якщо previousHash не збігається → виявлено форк
5. A → B: SYNC_REQUEST { fromIndex: 0, fullChain: true }
6. B надсилає весь ланцюг → A замінює свій якщо новий довший
```

## Криптографія та гаманець

### ECDSA P-256 (Web Crypto API)

- **Алгоритм:** ECDSA (Elliptic Curve Digital Signature Algorithm)
- **Крива:** P-256 (secp256r1, 256-біт)
- **Хешування:** SHA-256 (також double SHA-256 для деяких операцій)
- **API:** Вбудований `crypto.subtle` (жоден зовнішній пакет)

### Створення гаманця

```
1. Генерація 16 байтів ентропії (crypto.getRandomValues)
2. Створення 12-слівної seed-фрази (зі списку 256 слів)
3. PBKDF2(SHA-256, seed, "diadem-p256-v1", 100000 ітерацій) → 32 байти
4. Побудова PKCS#8 DER-кодованого P-256 приватного ключа
5. crypto.subtle.importKey('pkcs8', ...) → обчислення публічного ключа
6. Адреса = '0x' + SHA-256(publicKey).slice(0, 40)
```

**Ключовий момент:** Seed-фраза → ключі виводяться **детерміновано**. Одна й та сама фраза завжди дає одні й ті самі ключі. Це дозволяє відновлювати гаманець на будь-якому пристрої.

### Способи автентифікації

| Спосіб | Опис |
|--------|------|
| **Seed-фраза** | 12 слів → детерміновано відновлює ключі |
| **Ключ доступу** | `DDM1-...` base64-кодований експорт (адреса + ключі + seed) |
| **QR-код** | QR з ключем доступу для сканування на іншому пристрої |

### Зберігання ключів (KeyStore)

```javascript
localStorage:
  diadem_wallet          // Поточний гаманець (адреса, ключі, seed)
  diadem_wallet_backups  // Резервні копії попередніх гаманців
  diadem_settings        // Налаштування (тема, мова)
  diadem_cas_cache       // Кеш CAS для офлайн-запуску
  diadem_state_cache     // Кеш WorldState для швидкого старту
```

**Приватний ключ ніколи не покидає пристрій.** Він зберігається лише в `localStorage` браузера.

## P2P мережа

### Три рівні з'єднання

```
┌─────────────────────────────────────────────────┐
│ Рівень 1: WebRTC DataChannel (прямий P2P)       │
│ ├── Пряме з'єднання між браузерами              │
│ ├── Шифрований канал даних                       │
│ ├── Chunking для повідомлень >200KB              │
│ └── ICE сервери: stun.l.google.com              │
├─────────────────────────────────────────────────┤
│ Рівень 2: BroadcastChannel (одне пристрій)      │
│ ├── Комунікація між вкладками одного браузера    │
│ ├── Миттєва доставка, без сервера               │
│ └── Автоматичне виявлення пірів                  │
├─────────────────────────────────────────────────┤
│ Рівень 3: WebSocket Relay (через сервер)         │
│ ├── Фолбек коли WebRTC недоступний (SES lockdown)│
│ ├── Сигналінг для встановлення WebRTC            │
│ └── Сервер-нода зберігає та роздає стан          │
└─────────────────────────────────────────────────┘
```

### Протокол повідомлень

| Тип | Напрямок | Опис |
|-----|----------|------|
| `hello` | ↔ | Оголошення ноди + висота ланцюга |
| `sync_request` | → | Запит блоків починаючи з індексу |
| `sync_response` | ← | Відповідь з блоками |
| `new_block` | ↔ | Трансляція нового блоку |
| `new_tx` | ↔ | Трансляція нової транзакції |
| `state_sync` | ↔ | Повний обмін WorldState |
| `dm_instant` | ↔ | Миттєва доставка DM (до запису в блок) |
| `dm_typing` | ↔ | Індикатор набору (ефемерний) |

### Процес з'єднання

```
Браузер A                 Сервер              Браузер B
    │                       │                      │
    ├── announce ──────────→│                      │
    │                       ├── peers ────────────→│
    │                       │←── announce ─────────┤
    │←── peer_joined ───────┤                      │
    │                       │                      │
    ├── offer ─────────────→│── offer ────────────→│
    │                       │←── answer ───────────┤
    │←── answer ────────────┤                      │
    │                       │                      │
    ├── ICE candidates ────→│── ICE candidates ──→│
    │←── ICE candidates ────┤←── ICE candidates ──┤
    │                       │                      │
    │◄═══════ WebRTC DataChannel відкритий ═══════►│
    │                       │                      │
    ├── HELLO ────────────────────────────────────→│
    │←── HELLO ────────────────────────────────────┤
    │←── SYNC_RESPONSE ────────────────────────────┤
    │←── STATE_SYNC ───────────────────────────────┤
```

### Унікальні ID екземплярів

Кожна вкладка браузера отримує унікальний `nodeId`:

## Сервер-нода

Сервер — це не просто статичний хостинг. Він виконує три ролі:

### 1. HTTP-сервер
Роздає статичні файли (index.html, CSS, JS). SPA fallback для hash-роутингу.

### 2. WebSocket сигналінг
Релеїть WebRTC offer/answer/ICE між браузерами для встановлення прямих з'єднань.

### 3. Персистентна нода (Bootstrap Node)

```
┌─────────────────────────────────────────────┐
│              Сервер-нода                     │
│                                             │
│  ┌─────────────────┐  ┌─────────────────┐  │
│  │  В пам'яті:     │  │  На диску:      │  │
│  │  serverState    │  │  data/          │  │
│  │  (WorldState)   │  │  server-state   │  │
│  │                 │  │  .json          │  │
│  └────────┬────────┘  └────────┬────────┘  │
│           │ Синхронізація кожні 5с          │
│           └────────────────────┘            │
│                                             │
│  При підключенні нового піра:               │
│  → Надсилає збережений WorldState           │
│  → Пір одразу отримує всі дані              │
│                                             │
│  При отриманні broadcast від піра:          │
│  → Зберігає state_sync якщо вищий height    │
│  → Передає всім іншим пірам                 │
└─────────────────────────────────────────────┘
```

**Навіщо це потрібно?** Якщо всі браузери закриються і потім один відкриється — він отримає останній стан від серверної ноди. Дані не зникають.

## Контентно-адресоване сховище (CAS)

```
Дані → SHA-256(дані) → хеш
Зберігання: хеш → дані

Приклад:
  cas.put({ content: "Привіт!", author: "0x123..." })
  → hash: "a1b2c3d4..."

  cas.get("a1b2c3d4...")
  → { content: "Привіт!", author: "0x123..." }
```

### Властивості
- **Контентно-адресоване** — хеш = адреса даних
- **Іммутабельне** — одного разу записане не змінюється
- **Дедуплікація** — однакові дані зберігаються лише раз
- **Pinning** — важливі дані позначаються і не видаляються при GC
- **Реплікація** — піри обмінюються об'єктами (HAVE/WANT/BLOCK)

### Використання
- Пости → контент у CAS, хеш у транзакції
- Профілі → дані профілю у CAS, хеш on-chain
- Медіа → зображення у CAS
- Блоки → опціонально кешуються для офлайн-запуску

## IPFS Bridge

Опціональна інтеграція з IPFS для додаткової стійкості даних:

```
CAS (локальне) ←→ IPFS Bridge ←→ IPFS (глобальне)
     SHA-256               CID (base32)
```

**Режими:**
1. **Локальна IPFS нода** (localhost:5001) — додавання та пінінг файлів
2. **Публічні шлюзи** — ipfs.io, dweb.link, cloudflare-ipfs.com, gateway.pinata.cloud

Якщо IPFS недоступний — CAS працює автономно, дані синхронізуються через P2P.

## Соціальні функції

### Пости
- Текст + медіа (зображення, декілька файлів)
- Спойлер для медіа
- Збереження контенту в CAS, хешу в блокчейні
- Вартість створення: **1 DDM**
- Кожен лайк приносить автору **0.1 DDM**

### Сторіс
- Зображення + текст з кастомним стилем
- Налаштовувана тривалість (за замовчуванням 24 години, або перманентно)
- Перегляд у повноекранному режимі з прогрес-баром
- Можна приховати або **видалити** (on-chain транзакція)
- Архів прострочених сторіс на профілі

### Прямі повідомлення (DM)
- **Миттєва доставка** через P2P (ще до запису в блок)
- Текст + зображення
- Переказ DDM прямо у чаті
- Індикатор набору тексту (ефемерний, не зберігається)
- Дедуплікація: повідомлення не дублюється коли приходить і через P2P і через блок

### Лайки та реакції
- Лайк/анлайк на пости та відповіді
- Емодзі-реакції (окремо від лайків)
- Лайк = нагорода автору 0.1 DDM + репутація

### Підписки
- Підписатися/відписатися від користувачів
- Стрічка складається з постів підписок + рекомендованих
- Статистика: підписники/підписки

### Відповіді (Replies)
- Безкоштовні коментарі під постами
- Видалення своїх відповідей
- Лайки на відповіді

### Репости
- Поширення чужих постів у своїй стрічці
- Toggle (репост/скасувати)

### Збережені повідомлення
- Приватні нотатки (як "Збережене" в Telegram)
- Тільки для власника гаманця
- On-chain але не в публічній стрічці

## Токеноміка DDM

### Параметри

| Параметр | Значення |
|----------|----------|
| Загальна емісія (genesis) | 100,000,000 DDM |
| Faucet (тестнет) | 10,000 DDM для нових гаманців |
| Вартість посту | 1 DDM |
| Оновлення профілю | 0.5 DDM |
| Нагорода за лайк | 0.1 DDM (автору) |
| Нагорода валідатору | 10 DDM за блок |
| Стейкінг APY | 14.2% річних |
| Мінімальний стейк | 100 DDM |

### Потік токенів

```
Faucet (10k DDM) → Нові користувачі
        │
        ├─→ Створення постів (-1 DDM) → Burn
        ├─→ Оновлення профілю (-0.5 DDM) → Burn
        ├─→ Стейкінг → Нагороди валідатора (+10 DDM/блок)
        ├─→ Переказ → Інший гаманець
        └─→ Купівля декорацій → Burn

Лайки → +0.1 DDM автору (із системи)
```

## Proof of Stake консенсус

### Як працює

```
1. Отримуємо всіх валідаторів (стейк ≥ 100 DDM)
2. Рахуємо загальний стейк всіх валідаторів
3. Детермінований seed = SHA-256(previousBlock.hash + height)
4. target = seed % totalStake (зважений вибір)
5. Проходимо по валідаторах, віднімаємо стейк
6. Перший, де target < 0 — він продукує наступний блок
```

### Параметри

```
Інтервал блоку:        ~10 секунд
Макс TX в блоці:       100
Нагорода за блок:      10 DDM
Мін. стейк валідатора: 100 DDM
APY стейкінгу:         14.2% (compound daily)
```

**Детермінізм:** Той самий previousHash + height завжди обирає того самого валідатора. Це запобігає конфліктам.

## Безпека

### Захист приватних ключів
- Ключі зберігаються **тільки** в `localStorage` браузера
- Ніколи не передаються по мережі
- Ніколи не відправляються на сервер
- Seed-фраза для відновлення (справжня децентралізація)

### Підписання транзакцій
- Кожна дія потребує ECDSA підпис
- Верифікація підпису на кожній ноді
- Адреса виводиться з публічного ключа

### Захист від повтору (Replay Protection)
- `processedTxs` — множина оброблених хешів транзакцій
- `deletedPosts` — множина видалених постів (запобігає повторному додаванню)

### Форк-резолюція
- Довший ланцюг виграє
- Повна синхронізація при виявленні форку
- Детермінований вибір валідатора запобігає довільним форкам

### Цілісність даних
- CAS: SHA-256 верифікація при зчитуванні
- Блоки: перевірка previousHash
- Транзакції: верифікація ECDSA підпису

### SES Lockdown Protection
- Раннє збереження `window.__RTCPeerConnection` до того як MetaMask/SES заблокує API
- Fallback ланцюг: WebRTC → BroadcastChannel → WebSocket relay

## Технологічний стек

### Фронтенд (0 залежностей)
- **JavaScript** (ES Modules) — без React, без Vue, без фреймворків
- **Web Crypto API** — ECDSA P-256, SHA-256, PBKDF2
- **WebRTC DataChannels** — прямий P2P між браузерами
- **BroadcastChannel API** — синхронізація вкладок
- **Canvas API** — обрізка зображень, редактор сторіс
- **localStorage** — мінімальне збереження (тільки ключі)

### Бекенд (1 залежність)
- **Node.js** — HTTP + WebSocket сервер
- **ws@8.18.0** — WebSocket бібліотека
- **ES6 Modules** — import/export

### Протоколи
- **WebRTC ICE** (STUN: stun.l.google.com)
- **WebSocket** (сигналінг + relay)
- **IPFS HTTP API** (опціонально)
- **Кастомний P2P протокол** (HELLO, SYNC, STATE_SYNC, ...)

### Збереження
- Блокчейн — в оперативній пам'яті (відновлюється з пірів)
- Гаманець — localStorage (ніколи не надсилається нікуди)
- CAS — в оперативній пам'яті + кеш в localStorage
- Стан сервера — `data/server-state.json` (на диску)
